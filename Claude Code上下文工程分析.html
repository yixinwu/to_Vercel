核心观点

Agent能否有满意的交付，关键就在于能否理解上下文。这既考验模型的能力，也考验Agent系统提供准确充足上下文的能力。Claude Code的成功不仅源于Claude系列模型的强大，更有力的上下文支持是其关键所在。

一、上下文工程概述

1.1 什么是上下文？

如果把prompt当成一种指令、一次对话，那么上下文就是为每次对话提供足够的语境支持。

类比说明

就像两个人聊天，如果对方不知道你刚才在做什么、在想什么、有什么背景，就很难理解你的话。

例如：你说"帮我看看这个"，如果对方不知道：

  • 你正在写代码
  • 你遇到了什么错误
  • 你的项目结构是什么

他就无法给出真正有用的帮助。

1.2 从提示词工程到上下文工程

一个新的名词的出现往往是思维范式上的转变。一个工程的出现往往是为了解决某些问题产生的一种方法论。

Claude Code一次重构任务面临的问题：

上下文挑战 具体表现
上下文规模庞大 一轮Agent Loop产生 84,247 字节的上下文
多轮循环累积 一次重构任务需要多轮Agent Loop
多Agent协调 主Agent + SubAgent各有独立上下文
信息传递复杂 需要协调不同Agent之间的上下文传递

二、上下文工程的四大痛点

Drew Breunig 将上下文难题总结成四大痛点：

2.1 上下文中毒

问题定义

当模型虚构了工具调用状态信息并被写入上下文时，在多轮循环中，模型可能会把错误信息误认为是事实反复引用，并执着于实现不可能或无关的目标。

后果：后续每一步决策都在错误基础上前行，越走越偏。

2.2 情境干扰（上下文过长）

问题定义

当上下文过长时，会导致：

  • 注意力资源被稀释
  • 过度关注上下文
  • 忽略训练获得的内容
  • 输出质量下降

研究发现（Gemini 2.5 技术报告）

随着上下文超过 10万个token，大模型倾向于多步骤生成推理的长上下文，而非用于检索的长上下文。

2.3 语境混淆（多余信息过多）

问题定义

上下文中多余的内容会干扰大模型的响应。

例如：配置了数十个MCP函数时，每次调用大模型要对数十个MCP做判断，调用工具决策的准确性会大幅下降。

伯克利研究报告（今年6月）

大模型在引入函数调用或使用工具时，几乎所有大语言模型的性能都会比原始文本生成任务有所下降。在多函数调用场景中，准确率显著低于单函数调用。

2.4 语境冲突（多个信息源）

问题定义

语境冲突常出现在多个Agent协作时。

例如：导航subAgentA说中山路往东，导航subAgentB说中山路要往西，大模型做决策时到底听谁的？

三、Claude Code 上下文结构六大模块

基于 690 行的 context_dump.log 分析，Claude Code 的上下文结构分为六个部分：

模块 内容 作用
1. 系统信息内容
  • 系统身份定位
  • 安全策略
  • 交互规范
  • 代码风格
  • 工具使用指南
  • 任务管理
  • 工作流程
  • 示例展示
 定义Agent的基本行为规范
2. 会话延续上下文
  • 之前的会话完整历史
  • 会话中断原因和状态
  • 八段式压缩上下文
 保持会话连贯性
3. 工具调用和反馈
  • 工具调用参数
  • 执行反馈
  • SubAgent信息
  • TodoList任务状态
 记录工具执行状态
4. 已验证的关键任务节点 基于工具调用、日志等信息汇总 确认已完成的任务
5. 用户请求 按时间顺序结构化用户最新请求 保持用户意图清晰
6. 系统信息
  • 用户近期操作
  • 打开/点击的文件
 感知用户当前状态

3.1 八段式压缩上下文

将超过阈值的上下文总结成九段（实际日志发现）：

  1. 用户的主要请求和意图
  2. 关键技术概念
  3. 相关的文件位置
  4. 出现的问题及其解决方案
  5. 问题解决的思路方法结果
  6. 所有用户消息的完整记录和时间线
  7. 待完成的任务（包含挂起的任务）
  8. 当前工作及下一步的结构化摘要
  9. 挂起任务的单独总结（压缩提示词模板中可选）

3.2 特殊标记机制

标记 用途
system-reminder 标记用户消息、工具调用信息，与其他信息区分
ismeta 标记用户消息，用于与系统消息区分

四、上下文更新机制

4.1 更新流程概览

整个机制以事件触发为起点，经过消息队列、上下文处理等流程，到最后API调用响应为终点。

用户操作/系统事件 → 事件分类 → WD5分发处理 → K2创建消息 
→ 消息队列 → nO主循环 → wU2压缩判断 → Ie1上下文注入 
→ API调用 → 响应处理

4.2 核心流程步骤

步骤 核心函数 功能
1. 事件触发 WD5 事件分发中心
2. 消息创建 K2 创建对应的提示词
3. 消息队列 - 排队等待处理
4. 主循环 nO 控制整体流程
5. 压缩判断 wU2 判断是否需要压缩
6. 上下文注入 Ie1 按需注入上下文

五、上下文更新的三大原则

1. 事件驱动

  • 触发机制：文件操作、工具执行、用户输入、目录变更等触发事件
  • 分发中心：WD5函数作为事件分发中心
  • 模板匹配：每个事件对应相应的提示词模板
  • 内存事件：内存紧张时触发紧急压缩机制

2. 实时响应

  • 即时创建：K2函数立即创建对应的提示词
  • 队列管理：提示词被塞到消息队列
  • 时效保证：每次API调用前重新注入最新上下文
  • 中断支持：主循环多处设置中断点支持实时交互

3. 按需注入

  • 精准注入：只在触发事件时才生成提示词，只注入需要的提示词
  • 减少混淆：避免多工具调用的语境混淆
  • 优先级：消息队列将 system-reminder 标记的消息放在最前面
  • 用户优先：对 ismate 用户消息特殊处理，确保用户信息不丢失
  • 条件检查：Ie1函数只在存在相关事件队列时才进行注入
  • 压缩支持：可通过多种方式触发结构化压缩

设计精妙之处

Claude Code 实现了用户操作 → 事件生成 → 消息队列 → 条件注入 → API调用的完整链条，确保上下文始终保持最新且最相关。

六、上下文压缩机制

6.1 压缩触发机制

Claude Code 上下文压缩机制的核心是 qH1 函数，有三种触发方式：

触发方式 触发条件 特点
1. 自动触发 主循环调用 wU2 判断达到 token 使用阈值 自动执行 qH1 函数压缩
2. 手动触发 用户输入 /compact 命令 用户自主选择压缩时机，可传递压缩要点
3. 系统触发 系统检测到内存压力超过阈值 触发更激进的紧急压缩策略

手动触发示例

/compact 请重点关注TypeScript代码变更

这样会传递给压缩提示词模板，在原有八段式要点基础上增加自定义压缩重点。

6.2 压缩流程三阶段

阶段一：分析

  • 消息分类：工具消息、用户消息、其他类型
  • 多维度分析：
    • 消息类型统计
    • 工具使用情况统计
    • 时间序列分析
    • 文件操作报告
    • 内容复杂度分析
  • Token计算：计算压缩空间，生成分析报告

阶段二：执行

  • 生成提示词：创建压缩提示词
  • 整合要点：整合外部传递的压缩要点
  • 模型调用：向专用压缩大模型上报
  • 等待响应：获取压缩结果

阶段三：上下文重建

  • 重要文件恢复：在token预算内，根据时间频次等指标恢复重要文件
  • 状态恢复：恢复Agent状态（todo工具信息、会话延续上下文、任务进度等）
  • 核心提示保护：重新注入系统提示词，避免被压缩
  • UI状态恢复：对UI状态做恢复
  • 存储更新：对存储的信息进行更新

核心创新

Claude Code 压缩流程最核心的是调用专用压缩大模型进行总结抽象式压缩，前期的消息准备和后期的上下文重建同样起到关键作用。

七、多Agent协调机制

7.1 Task工具架构

Claude Code 采用 Task工具作为一种子Agent协作模式。

串行共享Agent

todo + Agent Loop 的回调

  • 串行执行
  • 上下文共享

并行隔离Agent

Task工具

  • 并行执行
  • 上下文不共享

7.2 避免上下文冲突的三大机制

机制一：隔离机制

隔离维度 具体措施
执行环境隔离 独立会话ID、空消息历史
上下文继承控制 只继承必要环境信息，不继承对话历史
工具权限限制 限制可用工具，不继承Task等工具，防止递归调用
任务注入 将拆解后的任务注入到不同的TaskAgent里

机制二：状态管理

管理维度 具体措施
无状态设计 每次调用完全独立
单向通信 TaskAgent只向主Agent传递工具调用报告，不传递上下文信息
并发协调 UH1函数管理多Agent并发

机制三：结果处理

处理阶段 具体措施
独立收集 每个Agent结果独立存储
智能排序 按Agent索引确保结果顺序一致
智能合成 KN5函数创建新的TaskAgent专门合并结果
错误隔离 单个Agent的错误不影响其他Agent

7.3 智能结果合成（KN5函数）

合成结果的指令包括：

  1. 阅读所有Agent的响应
  2. 识别每个响应的核心观点
  3. 检测观点之间的冲突
  4. 基于任务目标进行优先级排序
  5. 生成能够协调所有观点的统一方案
  6. 保留重要的技术细节和代码示例
  7. 组织成结构化的输出格式

八、总结与启示

8.1 Claude Code 上下文工程的核心要素

核心要素 实现方式 解决的问题
结构化上下文 六大模块清晰分层 语境混淆
事件驱动更新 实时响应 + 按需注入 时效性、准确性
智能压缩 专用压缩模型 + 三阶段压缩 上下文过长、情境干扰
多Agent隔离 独立环境 + 单向通信 + 结果智能合成 语境冲突
关键节点验证 已验证任务节点记录 上下文中毒

8.2 关键设计思想

核心设计思想

  1. 精准控制：不是简单地堆砌上下文，而是精细化控制注入时机和内容
  2. 层次分明：六大模块层次清晰，各司其职
  3. 动态平衡：在充足上下文和精简上下文之间动态平衡
  4. 隔离协作：通过隔离确保独立，通过合成确保协调
  5. 容错机制：关键节点验证、错误隔离、状态恢复

8.3 对Agent开发的启示

上下文管理方面

  • 建立结构化的上下文组织方式
  • 实现按需注入而非一次性全量注入
  • 建立完善的压缩机制
  • 保留核心信息，去除噪音

多Agent协作方面

  • 采用隔离机制避免上下文污染
  • 单向通信减少混乱
  • 独立的错误处理
  • 智能的结果合成机制

系统设计方面

  • 事件驱动架构更灵活
  • 实时响应用户操作
  • 建立完整的流程控制机制
  • 关键状态持久化保存

工程实现方面

  • 专用压缩模型的重要性
  • 消息队列管理
  • 条件检查避免无效调用
  • 用户体验优化（如中断点支持）

核心结论

Claude Code 的上下文工程展示了一套完整的解决方案：

  • 从结构设计到更新机制

这不仅是技术实现，更是一套完整的工程方法论，为构建复杂Agent系统提供了重要的参考范式。
